# 心愿单最佳卖出方案模块说明 v1

状态：Implemented  
日期：2026-07-25

## 1. 模块定位

该模块从用户资产中找出一组可执行的卖出组合，用保守估价说明它能覆盖多少
心愿目标。核心原则是：

- 资产是否可卖必须由用户确认。
- 价格与组合由可追溯数据和确定性算法产生。
- AI 只解释结果，不定价、不预测成交概率、不修改组合。

## 2. 主流程

```text
打开心愿详情
  → POST /sell-plans/prepare
  → 服务端读取用户心愿和全部未售资产
  → 判定资产数据就绪状态
  → 仅使用已确认且估价有效的资产
  → 确定性算法计算组合
  → AI 解释；失败时使用确定性文案
  → 保存当日方案快照
```

移动端只提交 `wishlist_item_id`、本地日期和是否刷新行情。资产候选、价格和状态
均由服务端按认证用户重新读取，客户端不是权威数据源。

## 3. 数据就绪规则

| 状态 | 判定 | 是否进入组合 |
|---|---|---|
| `needs_confirmation` | 状态从未被用户明确确认 | 否 |
| `needs_valuation` | 已确认可卖，但没有完整估价 | 否 |
| `stale_valuation` | 已确认可卖，但估价超过 7 天 | 否 |
| `ready` | 状态为闲置/出售中，且确认、估价均有效 | 是 |
| `excluded` | 用户确认仍在使用 | 否 |

心愿页支持批量选择“还在用 / 已闲置 / 准备出售”。确认通过
`confirm_asset_sellability` RPC 写入，确保状态、确认时间和来源同时更新。

页面分别展示：

- **已确认可卖**：所有 `ready` 资产的保守估价合计。
- **待确认潜力**：未确认资产已有保守估价的合计。
- **今日组合**：确定性算法最终选中的资产及目标覆盖率。

因此，未确认资产不会被误报为“没有资产、预计 ¥0”。

## 4. 行情刷新与组合算法

- 用户主动刷新时，单次最多处理 5 件资产。
- 刷新顺序为：缺少估价 → 估价过期 → 当前估价仍有效。
- 市场候选先经过同款识别，再由统计逻辑生成估价区间。
- 组合金额优先使用 `price_low`，缺失时才使用中位估价。
- 达到目标时：优先资产数量最少，其次超出目标金额最少。
- 无法达到目标时：优先总金额最高，其次资产数量更少。
- 状态空间最多保留 20,000 个，避免资产较多时计算失控。

当前没有可靠的流通速度字段，因此算法暂未引入“成交速度权重”，也不会让 AI
臆测流通概率。

## 5. AI 解释与快照

`SellPlanExplanationWorkflow` 只接收已经计算好的目标、组合和就绪统计。
结构化输出中的资产 ID 必须与确定性组合完全一致，否则解释作废并回退到模板。

每日方案保存在 `sell_plan_snapshots`，主要记录：

- 目标金额、组合、预计总额和覆盖率；
- 输入指纹 `input_fingerprint`；
- 就绪统计 `readiness_counts`；
- 算法版本 `calculation_version`；
- 估价基准时间 `valuation_as_of`；
- AI 或确定性解释 `explanation`；
- 行情刷新失败数。

资产状态或估价变化后会重新生成指纹和当日方案。

## 6. 当前边界

- 当前按心愿的完整 `target_price` 计算；项目尚无按心愿归集的资金流水，因此
  暂未扣除“已攒金额”形成目标缺口。
- 刷新失败的资产只有在原估价仍有效时才可能继续进入组合。
- 方案只提供计算与解释，不会自动上架、出售资产或写入目标进度。
- `/sell-plans/recommend` 作为旧兼容接口保留；当前移动端使用权威编排接口
  `/sell-plans/prepare`。
