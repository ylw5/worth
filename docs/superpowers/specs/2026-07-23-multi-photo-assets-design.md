# 物品多图与重新解析设计

## 目标

每件物品支持 1–5 张照片。用户可在新建或编辑时拍摄、从相册选择、删除和设置封面；编辑照片时可选择是否用全部当前照片重新解析物品信息。

范围仅包括 Android 和 iOS。

## 数据模型

- 将 `assets.photo_path` 迁移为非空有序数组 `photo_paths`。
- 迁移时把每条旧数据的单个路径转换为单元素数组，不丢失现有照片。
- 数据库约束每件物品必须保留 1–5 个路径。
- 数组第一项是封面；资产列表和详情页主图均使用它。
- App 为全部路径生成签名 URL，并保持相同顺序。
- 不新增图片子表或额外媒体服务。

## 新建流程

1. 用户通过系统相机连续拍摄，或从系统相册一次选择多张照片。
2. 选择页展示缩略图，允许删除和将任意照片设为封面；总数不得超过 5。
3. 用户继续后上传全部照片，并把全部签名 URL 一次发送给解析接口。
4. AI 合并多张照片的信息，返回一份可编辑表单。
5. 用户确认后保存资产并自动估价。
6. 上传或解析失败时不创建资产，并清理本次已上传的照片。

## 编辑流程

- 编辑页顶部展示全部已有照片，并提供拍摄、相册添加、删除和设为封面操作。
- 始终至少保留一张照片。
- “重新解析物品信息”开关只在照片发生变化后显示，默认开启。
- 开启时，先上传新照片，再用全部当前照片解析；解析结果只更新本地表单，用户再次点击保存后才写入数据库。
- 关闭时，跳过解析，直接保存照片顺序和当前手动信息。
- 最终保存继续自动重新估价；估价失败仍保留已保存的信息和照片。
- 用户在最终保存前退出时，清理本次为预览上传的新照片。

## 一致性与失败处理

- 新照片上传成功但解析或数据库更新失败时，不删除任何旧照片。
- 数据库更新失败时清理本次新上传的照片，旧数据保持不变。
- 数据库更新成功后才删除用户移除的旧存储对象；清理失败不回滚已保存资产。
- 多图解析失败时保留当前表单和照片选择，显示错误并允许重试或关闭重新解析后保存。
- 设置封面只调整数组顺序，不复制文件。

## API

- `/analyze` 从单个 `image_url` 改为 `image_urls`，校验数量为 1–5。
- OpenAI 请求为每个 URL 添加一个 `input_image`，只返回一份合并后的物品信息。
- 估价接口与市场检索流程不变。

## 验证

- 数据迁移把旧单图完整转换为单元素数组，并拒绝 0 张或超过 5 张。
- 单元检查覆盖封面排序、照片数量限制和解析失败不覆盖已编辑表单。
- 服务端测试覆盖 1 张、5 张和非法数量，以及多图都进入 OpenAI 请求。
- 移动端 lint、TypeScript、Android bundle 和 iOS bundle 通过。
- 在 Android/iOS 上手动验证拍摄、相册多选、删除、设封面、重新解析预览和跳过解析保存。
