# 固定成色选项设计

## 目标

把资产的“成色”从自由文本改为固定选项。图片解析、首次确认和后续编辑共用同一组值，避免 AI 描述与用户输入不一致。

## 固定选项

按从好到差排列：

1. `全新未使用`
2. `几乎全新`
3. `轻微使用痕迹`
4. `明显使用痕迹`
5. `重度使用或有瑕疵`
6. `无法判断`

判断标准：

| 选项 | 标准 |
| --- | --- |
| 全新未使用 | 用户能够确认物品从未使用；AI 不能仅凭外观照片推断 |
| 几乎全新 | 外观完整，没有可见划痕、磨损或污渍 |
| 轻微使用痕迹 | 有少量细小划痕、轻微磨损或轻微污渍 |
| 明显使用痕迹 | 有多处可见划痕、磨损、掉色或污渍，但整体完整 |
| 重度使用或有瑕疵 | 有严重磨损、破损、缺角、变形或照片可见的明显风险 |
| 无法判断 | 照片模糊、遮挡或缺少判断成色所需的关键角度 |

## 数据流

- 服务端把 AI 识别结果中的 `condition` 定义为固定枚举。
- AI 只根据照片中可见的外观选择成色；不能确认时返回 `无法判断`，不得猜测“全新未使用”。
- 移动端领域类型使用相同枚举。
- 确认页和编辑页共用现有 `AssetFormFields`，将成色文本框替换为与分类一致的选择按钮。
- 保存和估价继续传递选中的字符串，不改变现有 API 字段名和数据库列。

## 旧数据

通过数据库迁移一次性处理已有数据：

- 已属于六个固定选项的值保持不变。
- 其他自由文本统一更新为 `无法判断`，不根据文字猜测对应等级。
- 清理完成后为 `condition` 增加检查约束，只允许六个固定值。

## 校验与错误处理

- 新建和编辑保存时，成色必须属于固定选项。
- AI 结构化输出由服务端枚举约束；不合法结果按现有解析失败流程处理。
- `无法判断` 是合法值，允许用户保存或手动改为其他选项。
- 数据库拒绝六个固定值之外的写入。

## 验证

- 服务端测试验证 AI schema 只接受六个固定值，并拒绝旧自由文本。
- 移动端最小测试验证固定选项及成员判断。
- 数据库迁移验证旧自由文本被更新为 `无法判断`，且约束生效。
- TypeScript 检查和现有服务端测试通过。
- 手动确认首次识别页与编辑页均只能选择固定值，不能输入自由文本。

## 非目标

- 不新增独立瑕疵描述字段。
- 不按资产分类维护不同成色体系。
- 不尝试把已有自由文本自动映射为具体成色。
