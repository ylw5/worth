# 单页增量识别与照片主体抠图设计

## 目标

把拍照导入与确认合并为一个页面。用户每添加一张照片，系统立即上传该照片、增量识别资产信息并生成主体抠图，随后自动更新同页表单。用户手动修改过的字段不再被后续 AI 结果覆盖。

每张照片都尝试生成抠图。资产卡片和导入页使用抠图结果，资产详情相册继续展示原图。

## 范围

本次包含：

- 拍照逐张添加并处理。
- 相册可一次多选，按选择顺序逐张排队处理。
- 每张照片独立上传、识别和抠图。
- 当前表单作为下一张照片的识别上下文。
- AI 自动更新未被用户手动修改的字段。
- 每张照片分别展示识别和抠图状态，并可重试失败项。
- 任意照片可设为封面，不重复抠图。
- 保存原图和对应的可选抠图路径。
- 删除独立确认页和跨页草稿状态。
- 编辑页在重新解析新照片时补做抠图。

本次不包含：

- 手工描边、擦除、恢复或更换背景。
- 后台任务队列、并行批处理或历史资产批量回填。
- 用抠图替代详情页原始照片。
- 让 AI 覆盖用户已经修改的字段。

## 单页交互

页面从上到下由照片区、资产表单、错误提示和“保存并估价”按钮组成。

### 添加照片

- 拍照每次加入一张。
- 相册保留多选，但多张照片只进入本地队列，不并行处理。
- 同一时间最多有一张照片处于处理中。
- 处理期间暂停继续添加、删除、排序和设为封面，防止照片状态与异步结果错位。
- 当前照片完成后自动处理队列中的下一张。

### 照片状态

每张照片分别记录识别与抠图状态：

- `pending`：等待处理。
- `processing`：正在处理。
- `succeeded`：处理成功。
- `failed`：处理失败，可单独重试。

照片缩略图优先显示抠图；没有抠图时显示原图。状态文案组合为：

- 两项成功：“已解析”。
- 识别成功、抠图失败：“抠图失败，可重试”。
- 识别失败、抠图成功：“解析失败，可重试”。
- 两项失败：“处理失败，可重试”。

重试只调用失败的能力，不重复已经成功的识别或抠图。

### 表单更新

- 第一张照片以空白表单作为上下文。
- 后续照片把当前表单传给识别服务，模型基于新照片补充或修正完整资产信息。
- 客户端维护用户修改字段集合。
- AI 结果只写入空字段和未被用户修改的字段。
- `name`、`brand`、`model`、`category`、`condition`、`search_query` 分别保护。
- 规格文本作为一个整体保护；用户一旦修改规格，后续 AI 不再覆盖整个规格集合。
- 购买日期和购买价格不由照片识别生成，只由用户编辑。
- 用户可在照片处理期间编辑表单；异步结果到达时仍遵守最新的保护集合。

### 保存

- 任一照片仍在排队或处理中时，禁止保存。
- 任一照片识别失败时，禁止保存，并提示先重试或删除该照片。
- 只有抠图失败不阻止保存，对应照片回退原图。
- 表单继续使用现有必填、购买日期和购买价格校验。
- 保存成功后清空本地处理状态，刷新资产列表并返回资产页。
- 用户未保存便离开时，删除本次已上传的所有原图和抠图。

## 服务端接口

识别与抠图保持两个独立能力，使失败重试不会重复成功的工作。

### 增量识别

现有 `/analyze` 继续接收 `image_urls`，单页导入每次只传当前照片的一个 URL，并新增可选的 `current_asset`。

服务端提示模型：

- 新照片属于 `current_asset` 所描述的同一件资产。
- 基于新证据返回完整的最新资产信息。
- 不猜测照片中无法确认的规格。

服务端不负责判断用户手动字段；字段保护只在客户端完成。

编辑页可继续传入多张 URL，并使用当前表单作为上下文。

### 主体抠图

新增 `/cutout`，请求只包含一个 Supabase 签名图片 URL，响应包含可选的透明 PNG Base64。

- 只允许下载与配置的 `SUPABASE_URL` 同源的 `http` 或 `https` URL。
- 下载上限为 10 MB。
- 使用 `rembg` CPU 后端。
- 抠图前按比例把最长边限制为 1024px。
- 输出必须是带透明通道且主体非空的 PNG。
- 下载、模型或图片处理失败时返回空结果，不影响识别。

## 客户端处理队列

每个 `AssetPhoto` 扩展以下状态：

- 原图本地 URI、Base64、Storage 路径和分析 URL。
- 可选的抠图 Storage 路径和签名 URL。
- 识别状态。
- 抠图状态。

处理一张照片的顺序：

1. 上传原图并取得签名 URL。
2. 同时请求增量识别和主体抠图。
3. 识别成功后按字段保护规则合并表单。
4. 抠图成功后立即上传透明 PNG。
5. 更新该照片的两个状态。
6. 继续处理队列中的下一张。

抠图上传失败等同于抠图失败；原图路径继续有效。

删除已经上传的未保存照片时，同时删除其原图和抠图。处理中禁止删除，因此无需取消飞行中的 HTTP 请求。

## 数据模型

`assets` 表新增：

- `photo_cutout_paths jsonb not null default '{}'::jsonb`

字段保存一个原图路径到抠图路径的对象：

```json
{
  "user/original-a.jpg": "user/cutout-a.png",
  "user/original-b.jpg": "user/cutout-b.png"
}
```

选择 JSON 对象而不是平行数组，原因是：

- 重排原图不需要同步另一个数组索引。
- 删除照片可直接按原图路径移除对应项。
- 抠图失败的照片可以自然地没有映射。
- 旧资产默认空对象并自动回退原图。

客户端读取资产时为每个抠图路径生成签名 URL，并形成同结构的 `photo_cutout_urls`。

资产卡片使用：

1. 当前第一张 `photo_paths[0]` 对应的抠图 URL；
2. 若不存在，则回退 `photo_urls[0]`。

详情相册始终只使用 `photo_urls`。

## 编辑页

- 现有原图加载时带上对应的抠图路径与 URL。
- 重新排序时，抠图跟随各自的原图对象，不重新生成。
- 删除照片后，保存成功再删除该原图及其抠图。
- 新添加照片在用户点击现有“重新解析照片”时上传、识别并抠图。
- 新照片的抠图失败不阻止编辑保存；识别失败沿用现有错误处理。
- 编辑保存时根据最终照片集合重建 `photo_cutout_paths`，不会保留已删除照片的映射。

## 路由与状态清理

删除：

- `mobile/src/app/confirm.tsx`
- `mobile/src/providers/draft-provider.tsx`
- 根布局中的 `DraftProvider`
- 根布局中的 `confirm` 路由
- 拍照页到 `/confirm` 的跳转

所有新增资产状态保留在拍照页组件内。保存或离开后的清理都由该页面负责。

## 错误处理

- 原图上传失败：该照片处理失败，不调用识别或抠图。
- 识别失败：保留照片和已有抠图，允许只重试识别。
- 抠图失败：保留识别结果和原图，允许只重试抠图。
- 抠图上传失败：视为抠图失败，并清理未能持久化的临时结果。
- 保存失败：保留页面、表单和已上传文件供用户直接重试。
- 保存成功前离开：清理本次已上传文件。
- 保存成功后估价失败：资产仍然有效，沿用现有非阻塞估价行为。

## 验证

服务端聚焦测试覆盖：

- 增量识别把当前资产作为上下文，并仍使用全部传入 URL。
- `/cutout` 只接受配置的 Supabase 同源 URL。
- 输入图片最长边被限制为 1024px。
- 成功返回 PNG Base64。
- 抠图异常返回空结果，不影响识别端点。

客户端最小逻辑测试覆盖：

- AI 合并不会覆盖用户保护字段。
- 未保护字段会被后续识别结果更新。
- 规格保护独立生效。
- 当前封面从“原图路径 → 抠图 URL”映射中正确选择并回退。

静态与交互验证：

- Node 测试、TypeScript、Expo lint 和 Python pytest 全部通过。
- 相册多选后严格逐张处理。
- 表单在每张识别完成后更新。
- 手动字段在后续照片完成后保持不变。
- 每张缩略图优先显示自己的抠图。
- 任意照片设为封面后，资产卡片使用其已有抠图。
- 详情相册仍展示原图。
- 离开未保存页面后 Storage 不保留暂存文件。

## 成功标准

- 导入与确认只存在于一个页面。
- 用户添加照片后无需点击批量解析按钮，表单自动增量更新。
- 每张照片独立尝试抠图，并能独立重试失败项。
- 用户手动输入不会被异步 AI 结果覆盖。
- 任意抠图失败都不会阻止有效资产保存。
- 原图和抠图文件在保存、删除和取消流程中均无错误关联或明显孤儿文件。
