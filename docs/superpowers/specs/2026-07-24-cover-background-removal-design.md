# 封面主体抠图设计

## 目标

拍照导入资产时，保留全部原图用于识别和详情展示，仅对用户选定的第一张封面图执行主体抠图。资产列表优先展示透明背景的主体封面，使卡片效果更接近商品缩略图；抠图失败不得阻塞识别、保存或估价。

## 范围

本次包含：

- 新增资产时只抠第一张封面图。
- 保留原图，不使用抠图结果进行 AI 识别。
- 确认页同时展示封面效果和原始照片。
- 列表卡片优先使用抠图封面，并在缺失时回退第一张原图。
- 编辑时更换或重排封面后，清除旧抠图并回退新的第一张原图。

本次不包含：

- 每张照片抠图。
- 手工描边、擦除、恢复或更换背景。
- 独立抠图页面、任务队列和后台重试。
- 为历史资产批量生成封面。
- 编辑资产时自动重新抠图。

## 技术方案

在现有 FastAPI 服务中使用 `rembg` 的 CPU 后端完成主体分割，生成带透明通道的 PNG。移动端不增加原生模块，iOS、Android 和 Web 共用现有 HTTP 与 Supabase Storage 数据流。

不使用生成式图片模型重绘物品，以免改变外观、颜色、磨损或配件细节。暂不接入第三方收费抠图 API，避免新增凭据、图片隐私边界和按量费用。

服务端复用一个惰性初始化的 `rembg` session，避免每次请求重复加载模型。模型初始化或处理失败时返回空的封面结果，不让 `/analyze` 整体失败。

## 数据模型

`assets` 表新增：

- `cover_photo_path text null`：透明 PNG 在现有私有 `asset-photos` bucket 中的路径。

现有 `photo_paths text[]` 语义保持不变，仍只保存原始照片。客户端 `Asset` 类型新增可空的 `cover_photo_path`，读取资产时为该路径创建短期签名 URL，并暴露可选的 `cover_photo_url`。

不把抠图文件插入 `photo_paths`，因此详情相册的照片数量和排序不会变化。

## 新增资产数据流

1. 用户添加 1–5 张照片，并通过现有排序或“设为封面”交互决定第一张。
2. 移动端把全部原图上传到 Supabase Storage。
3. 移动端调用 `/analyze`，传入全部原图的签名 URL。
4. 服务端使用全部 URL 完成资产识别，同时只下载并处理 `image_urls[0]`。
5. `/analyze` 返回识别字段和可选的 `cover_image_base64` 透明 PNG 数据。
6. 移动端收到 PNG 后立即上传到同一个私有 bucket，并把路径保存在草稿中。
7. 用户确认保存时，`createAsset` 同时写入原图路径和可选的封面路径。
8. 用户取消或未保存便离开确认页时，现有清理流程同时删除已暂存的原图和抠图文件；保存失败时保留暂存文件供原页重试。

PNG 只在 `/analyze` 响应和确认草稿中短暂存在；上传成功后草稿保存路径与预览 URL，不长期保留大段 Base64。

## 交互

### 拍照页

- 第一张照片继续标记为“封面”。
- 用户点击其他照片的“设为封面”后，它移动到第一位。
- 点击“解析照片”后，按钮显示“正在识别并生成封面…”，并禁用重复提交。
- 不增加单独的抠图开关或配置项。

### 确认页

- 顶部新增“封面效果”区域，在浅灰背景上使用 `contain` 展示透明主体。
- “原始照片”区域继续展示全部原图。
- 若没有生成抠图，“封面效果”展示第一张原图，并显示“未能分离主体，将使用原图”。
- 用户仍通过现有“保存并估价”按钮完成保存。

### 列表与详情

- 资产列表卡片优先使用 `cover_photo_url`，否则使用 `photo_urls[0]`。
- 卡片继续使用 `contain` 和现有中性背景。
- 资产详情相册只展示 `photo_urls`，不混入抠图文件。

### 编辑

- 编辑页仍展示和管理原始照片。
- 若原始照片顺序或集合变化，保存时把 `cover_photo_path` 清空，并在保存成功后删除旧抠图文件。
- 新封面暂时使用第一张原图；用户无需处理过期或错误的旧抠图。

## 错误与清理

- 抠图下载超时、模型加载失败、处理异常或没有可用结果：返回空封面，识别结果照常返回。
- 抠图上传失败：确认流程继续，草稿不记录封面路径，并使用第一张原图预览。
- 识别失败：沿用现有行为，删除本次已上传的原图和抠图。
- 用户取消确认：删除本次暂存的原图和抠图。
- 资产保存失败：保留当前确认页状态供重试；离开页面时执行清理。
- 编辑保存失败：数据库仍保留旧封面路径，不提前删除旧文件。
- 编辑保存成功：数据库清空旧封面路径后，再尽力删除旧抠图；删除失败不回滚有效资产。

## 验证

服务端增加聚焦测试：

- 成功时只下载和处理第一张 URL，并返回 PNG Base64。
- 多张照片仍全部传给现有识别逻辑。
- 抠图异常时识别结果正常返回，封面字段为空。
- 非 PNG 或空结果不会作为有效封面返回。

客户端验证：

- 类型检查和 lint 通过。
- 新增资产时可看到抠图预览并保存。
- 卡片优先展示透明封面，详情仍展示原图。
- 抠图失败时确认页和卡片回退第一张原图。
- 取消确认后，原图和抠图暂存文件均被删除。
- 编辑更换封面后不再显示旧抠图。

## 成功标准

- 新增资产的列表卡片能呈现与参考图相近的独立物品主体效果。
- 只有用户提交时排在第一位的照片被抠图。
- 原图、识别准确性和详情相册行为不受影响。
- 任何抠图故障都不会阻止资产被识别和保存。
