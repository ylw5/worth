# Worth 双视图残值预测与后台行情设计

## 目标

在物品详情页增加“年化持有成本”和“今日行情”两个可切换视图。系统定期在后台采集市场数据、调用 AI 补充公开证据，并使用可复算的统计逻辑生成当前残值、6 个月和 12 个月预测区间。

功能覆盖所有现有物品分类。AI 负责识别、查询规划、证据抽取、候选匹配和结果解释，不直接决定最终价格。

## 产品原则

- 当前行情、未来预测、官方回收报价和新品价格必须使用不同名称，不混称为成交价。
- 每个金额都能追溯到采集时间、来源、样本量和计算版本。
- 数据不足时只展示已知事实，不为了完整而生成低可信预测。
- 页面只做量化比较，不输出“应该卖”“应该换新”等主观建议。
- 后台失败不阻塞详情页，不覆盖上一次成功结果。
- 继续使用 Supabase 作为业务数据源，不为 Cloudflare 重复建立 D1 数据库。

## 用户体验

### 页面位置

双视图卡位于物品照片之后、基础资料之前。现有底部“价格历史”继续保留，作为逐次估价的可核对记录。

### 顶部切换

卡片顶部使用两段式切换：

- 年化持有成本
- 今日行情

默认展示上次选择；没有购买日期或购买价格时默认展示“今日行情”。

切换使用两个 `Pressable`，提供选中状态、按钮语义、足够触控面积和屏幕阅读器标签。不引入新的 UI 组件库。

### 年化持有成本

参考 UI 的视觉层级：

1. 顶部显示“平均每天花掉”。
2. 主金额显示截至今天的平均每日折旧。
3. 次级信息显示累计折旧、年化持有成本和持有天数。
4. 只有配置了具体换新目标时，才显示与换新方案的量化差额。
5. 下方展示残值曲线和预测区间。
6. 底部显示数据来源、预测日期和置信度。

计算口径：

```text
累计折旧 = 买入价格 - 当前残值
平均每日折旧 = 累计折旧 / 持有天数
年化持有成本 = 累计折旧 / 持有天数 × 365.25
继续持有 H 月的预计成本 = 当前残值 - H 月预测残值
换新 H 月的预计成本 = 换新买入价 - 换新物品 H 月预测残值
```

当前残值高于买入价时显示“平均每天增值”和“年化增值”，不将结果截断为零。

缺少买入价、买入日期或当前残值时，不显示计算结果，改为明确列出需要补充的字段。持有不足 30 天时仍显示累计变化，但提示年化结果波动较大。

残值曲线包含：

- 买入时点。
- 今天的当前残值。
- 6 个月预测点。
- 12 个月预测点。
- 每个未来点的上下区间。

第一版使用 React Native `View` 绘制固定尺寸的点和线段，不引入完整图表库，不提供缩放、拖动或日内走势。

### 今日行情

今日行情展示：

- 今日参考市价。
- 第 25–75 百分位参考区间。
- 匹配样本数。
- 相比上一个不同日期行情点的涨跌额和涨跌比例。
- 30 天、90 天和全部三个时间范围。
- 每日行情曲线。
- 数据更新时间和来源说明。

只有一个行情点时显示“行情积累中”，不显示 `0%`。没有今天的成功快照时继续展示上次价格，并标记“更新中”或“数据截至 YYYY-MM-DD”。

页面文案统一使用“参考市价”“挂牌样本”“预测区间”，不使用“实时成交价”。

### 后台任务状态

详情页不轮询单个 Workflow。App 读取 Supabase 中的最新运行状态：

- `queued`：等待后台分析。
- `running`：行情更新中。
- `succeeded`：显示最近完成时间。
- `failed`：显示“本次更新失败，仍展示上次结果”。

页面刷新只重新读取 Supabase，不直接启动 AI 深度分析。资产资料修改后，数据库将对应行情标记为待分析，由下一个调度周期优先处理。

## 全品类估值资料

所有品类共享品牌、型号、规格、成色、购买日期、购买价格和搜索词。品类特有信息存入 `assets.valuation_profile`。

| 品类 | 主要价值因素 |
| --- | --- |
| 数码 | 容量、电池健康、维修、保修、附件、发布时间 |
| 家电 | 使用年限、能效、功能、安装拆卸成本、保修 |
| 家具 | 品牌、材质、尺寸、运输成本、所在地 |
| 服饰箱包 | 品牌、款式、尺码、季节、真伪、磨损 |
| 珠宝腕表 | 型号、材质、重量、证书、附件、保养记录 |
| 收藏 | 年代、版本、发行量、品相、来源、公开成交记录 |
| 交通工具 | 年份、里程、事故、过户、地区、维修记录 |
| 其他 | 品牌、型号、使用年限、成色、市场样本 |

`valuation_profile` 结构：

```json
{
  "schema_version": 1,
  "attributes": {
    "battery_health": 88,
    "repair_history": "无",
    "warranty_until": "2027-01-10"
  },
  "confirmed_keys": ["battery_health", "repair_history"]
}
```

AI 可以新增或更新未确认字段，不能覆盖 `confirmed_keys` 中的用户确认值。

## 数据来源

### 当前市场挂牌

沿用现有闲鱼搜索：

1. 使用用户确认的 `search_query` 搜索前三页。
2. 按商品 ID 去重。
3. AI 排除配件、广告、错误型号和关键规格不一致的候选。
4. 至少 5 个匹配样本才计算中位数和四分位区间。

该接口需要服务端登录 Cookie，属于需要登录态的内部搜索能力。部署前必须在 Cloudflare Worker 环境执行真实采样验证。若网络环境或风控导致失败，保留上次快照，不生成伪造的当日行情。

### AI 联网证据

通过 Cloudflare AI Gateway 调用 OpenAI Responses API Web Search，获取：

- 官方首发价和当前售价。
- 产品发布时间和换代事件。
- 官方换新计划及适用条件。
- 公开拍卖、成交或行业资料。
- 同系列相邻代际的当前价格。

AI 输出结构化证据，不把自然语言总结直接交给估值公式。

### 官方换新报价

官方换新通常需要型号、序列号、设备状态问卷和实物检测。第一版不自动提交 IMEI、序列号等敏感信息。

用户可以手动保存一次报价：

```text
来源
报价
报价日期
有效期
适用条件
```

官方回收价作为独立处置渠道，不等同于个人二手成交价。

### 自有真实成交

后续允许用户把物品标记为已出售并记录：

- 最后挂牌价。
- 实际成交价。
- 成交日期。
- 出售渠道。
- 成交耗时。

真实成交数据用于回测挂牌估值与可实现售价之间的差异，不在第一版自动公开或跨用户共享。

## 标准化证据

每条价格证据统一为：

```text
source_type: listing | sold | trade_in | retail | auction
source_name
title
price
currency
observed_at
condition
product_match
region
url
confidence
```

价格进入计算前必须满足：

- 币种可识别并转换为人民币。
- 观察时间存在。
- 价格为正数。
- 物品匹配通过。
- 来源类型明确。

完整网页正文、访问令牌、Cookie 和个人身份信息不写入数据库或日志。

## 预测方法

### 方法选择

预测引擎按以下顺序选择第一种满足条件的方法：

1. 同款自身时间序列。
2. 同系列跨代际截面曲线。
3. 同品类可比物曲线。
4. 证据不足，不输出未来残值。

AI 不直接输出最终价格，也不直接输出最终置信度。

### 自身时间序列

启用条件：

- 至少 4 个不同日期的有效行情。
- 历史覆盖至少 21 天。
- 每个行情点至少 5 个匹配样本。

计算：

1. 使用每日挂牌中位数。
2. 用每日四分位区间限制异常变化。
3. 对 `log(price)` 和日期执行线性回归。
4. 外推 6 个月和 12 个月。
5. 根据历史残差与预测周期生成区间。

使用对数价格避免预测为负数。外推结果不得低于零，也不得在没有升值证据时超过当前价格。

### 跨代际截面曲线

自身历史不足时，AI 搜索同品牌同系列的当前和旧型号，提取发布时间、首发价和当前二手参考价。程序按上市年龄拟合价格保持率，再将当前物品映射到曲线。

### 同品类可比物

没有明确型号代际时，按品类价值因素选择可比物。AI 只负责提取和匹配，程序使用通过校验的价格、年龄和成色计算。

### 个体修正

当前市场基线生成后，再根据用户确认的 `valuation_profile` 进行个体修正。修正系数必须来自已积累的真实成交或同款分层样本；没有数据支撑的字段只降低置信度，不凭规则硬编码折价比例。

## 置信度

置信度由程序计算：

```text
样本数量         30%
历史覆盖天数     25%
来源一致性       20%
规格匹配完整度   15%
数据新鲜度       10%
```

等级：

- 高：80–100。
- 中：60–79。
- 低：40–59。
- 不足：0–39，不输出未来残值。

12 个月预测区间必须宽于或等于 6 个月区间。页面同时展示分数、等级和主要缺失证据。

## Cloudflare 后台架构

### 部署边界

第一阶段只把市场采样、AI 联网分析和残值预测部署到 Cloudflare。现有 `rembg`、ONNX Runtime 和图片抠图保留在当前 Python 服务。

后台使用：

- Cloudflare Cron 定时触发。
- Cloudflare Python Workflows 执行持久步骤和重试。
- Cloudflare Python Worker 提供健康检查和受保护的任务查询入口。
- Cloudflare AI Gateway 调用 OpenAI Responses API。
- Supabase 保存业务状态和结果。

### 调度频率

- 每日：当前市场采样和每日行情快照。
- 每周：AI 联网深度研究及 6/12 个月残值预测。
- 资产信息修改后：标记为待分析，在下一个周期优先执行。
- 每月：使用已积累历史重新校准品类和代际曲线。

Cron 使用 UTC 配置，产品文档同时标注对应的 Asia/Shanghai 时间。

### Workflow 步骤

每个规范化 `market_key` 创建一个 Workflow：

1. 原子领取任务。
2. 读取共享该行情的资产。
3. 采集当前市场样本。
4. AI 匹配候选。
5. 按调度类型执行联网研究。
6. 标准化并校验证据。
7. 写入每日行情快照。
8. 生成市场级预测。
9. 叠加每件资产的个体修正。
10. 写入预测并完成任务。

每个外部数据源是独立 Workflow 步骤。可重试错误使用指数退避；认证失效、输入无效和来源明确拒绝属于不可重试错误。

### Cloudflare 兼容改造

Cloudflare Python Workers 只支持异步 HTTP 客户端。现有同步 `requests` 调用改为 `httpx.AsyncClient`：

- Supabase 鉴权请求。
- 闲鱼搜索请求。
- 图片下载以外的新后台请求。

估值纯函数继续复用。抠图代码不进入 Worker 包，避免 `rembg`、ONNX Runtime 和模型体积超出 Worker 运行环境。

### 机密

以下配置使用 Cloudflare Secrets：

- Supabase URL。
- Supabase service role key。
- 闲鱼 Cookie。
- Cloudflare AI Gateway 凭据。
- OpenAI 或统一结算凭据。

服务角色密钥只存在后台 Worker。移动端继续使用 Supabase 用户会话和 RLS。

### RLS 与后台写入

- 用户只能读取自己的行情、运行记录和预测。
- 移动端不能创建或修改后台运行记录。
- 后台使用 service role 完成跨表写入。
- 数据库函数负责原子领取任务和完成任务，防止两个 Workflow 重复处理同一 `market_key`。

## 错误与降级

- 闲鱼失败：不写当日行情，继续保留上次结果。
- Web Search 失败：允许用已有时间序列预测，并降低置信度。
- AI 匹配失败：不把未筛选样本写入行情。
- 样本不足：记录运行成功但数据不足，不写价格快照。
- 预测失败：行情仍可展示，预测区域显示暂无可靠预测。
- Workflow 最终失败：状态设为 `failed`，下次定时任务可以重新排队。
- Cookie 失效：使用固定错误码，禁止在错误信息中包含 Cookie。
- 页面任何错误都不得把上次成功金额重置为空。

## 性能与成本

- 以 `market_key` 而不是资产 ID 调度，同款资产共享一次采样和研究。
- 每日任务不执行 Web Search；Web Search 默认每周一次。
- 同一来源和同一 URL 在有效期内不重复研究。
- App 只从 Supabase 读取，不直接等待 Workflow。
- 不引入 D1、KV、Queues、Durable Objects 或独立 Agent 框架；出现明确吞吐瓶颈后再增加。

## 验证

### 纯函数

- 持有天数、每日折旧和年化成本。
- 当前价格高于买入价时的增值结果。
- 6/12 个月持有成本。
- 行情按日期去重和时间范围过滤。
- 对数趋势拟合和非负预测。
- 置信度边界。
- 数据不足时拒绝预测。

### 数据库

- 同一用户、行情和日期只能有一个快照。
- 两个 Workflow 不能同时领取同一任务。
- 同一行情的多个资产共享市场快照。
- 不同用户和不同搜索词不会串数据。
- 移动端不能写后台任务。
- 后台失败不会覆盖最近成功价格。

### Worker 与 Workflow

- Cron 能创建到期任务。
- 每个步骤可以独立重试。
- 不可重试错误立即终止。
- 闲鱼 Cookie 不出现在日志。
- Cloudflare 真实环境能完成一次闲鱼采样。
- AI Gateway 能返回带网页引用的结构化证据。
- Workflow 重启后从已完成步骤继续。

### 移动端

- 两个视图可切换并具有无障碍状态。
- 零个、一个和多个行情点均可展示。
- 缺购买信息时自动进入今日行情。
- `queued`、`running`、`failed` 和过期数据状态有明确文案。
- 预测区间和置信度始终同时展示。
- 参考图中的日均成本、残值曲线和数据来源层级在手机宽度下成立。

### 回测

每月选择已经到达预测目标日期的历史记录：

- 比较预测值与当日市场快照。
- 记录绝对百分比误差。
- 统计真实值落入预测区间的比例。
- 按品类、方法和置信度分组。
- 不自动根据少量回测结果修改生产参数。

## 分阶段交付

### 第一阶段：可审计行情

- 每日 `market_snapshots`。
- Cloudflare Cron 与 Workflow。
- 任务状态。
- 今日行情视图。

### 第二阶段：持有成本

- 购买数据计算。
- 双视图切换。
- 日均和年化持有成本。
- 无预测时的残值曲线。

### 第三阶段：残值预测

- AI 联网证据。
- 6/12 个月预测。
- 置信度和个体修正。
- 后台预测状态。

### 第四阶段：换新比较与回测

- 用户选择具体换新目标。
- 官方报价录入。
- 持有与换新的量化对比。
- 实际出售记录和预测回测。

每一阶段都能独立运行和验证。后续阶段失败不影响前一阶段的已有行情能力。

## 完成标准

- 所有现有品类都能进入统一后台分析流程。
- 详情页能切换年化持有成本与今日行情。
- 页面不等待 AI 或市场采样。
- 当前行情、预测和换新报价来源明确。
- 数据不足时不会生成伪精确结果。
- Cloudflare Workflow 能可靠重试并保留任务状态。
- 上次成功结果在后台失败时继续可用。
- 预测可以用后续行情和真实成交数据回测。
