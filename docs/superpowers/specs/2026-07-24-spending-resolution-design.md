# 忍住消费记录设计

## 目标

在购物评估对话中，Agent 可以在信息充分时建议用户放弃本次购买。只有当商品价格明确且用户点击“确认不买”后，本次金额才计入该用户的“忍住消费”总额。

本期只实现对话内的建议、确认和记录能力，不决定总额未来展示在哪个页面。

## 产品规则

- Agent 仅在结论为“不买”且金额明确时生成忍住消费动作。
- 商品价格未知时，Agent 每轮最多追问一个问题；取得明确的正数金额前不生成动作。
- 用户点击“确认不买”后立即确认，不设置 24 小时冷静期。
- 一次购物评估对话最多有一条忍住记录。
- 未确认记录跟随 Agent 的最新结论：
  - 再次建议不买时，可以更新为最新金额和商品快照。
  - 后续改为建议买时，撤销未确认记录。
- 已确认记录不可修改或撤销，也不受后续对话结论变化影响。
- 重复点击、请求重试和多设备并发不得重复计入金额。

## 数据模型

新增 `public.spending_resolutions`：

| 字段 | 类型 | 约束与用途 |
| --- | --- | --- |
| `id` | `uuid` | 主键，默认生成 |
| `user_id` | `uuid` | 必填，关联 `auth.users`，受 RLS 保护 |
| `evaluation_id` | `uuid` | 必填，关联 `purchase_evaluations`，唯一 |
| `amount` | `numeric(12, 2)` | 必填且大于 0 |
| `product_snapshot` | `jsonb` | 必填，保存确认候选对应的待购商品快照 |
| `image_paths` | `text[]` | 必填，默认空数组，保存私有存储路径 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 最近一次更新待确认记录的时间 |
| `confirmed_at` | `timestamptz` | 为空表示待确认，有值表示已计入 |

`product_snapshot` 复用当前评估商品结构，包含标题、链接、来源类型、原始描述、分类和子分类。图片只保存私有存储路径，不保存会过期的签名 URL。

`evaluation_id` 使用默认的删除限制；存在忍住记录时不能直接删除关联对话。本期没有删除聊天功能，未来若增加删除能力，再单独定义已确认记录的保留策略。

当前客户端已读写 `purchase_evaluations.decision`，但最新迁移删除了该字段。新迁移恢复该字段，默认值为 `pending`，并限制为 `pending`、`buy` 或 `skip`，使现有决策链路与数据库重新一致。

“忍住消费”总额按需计算：

```sql
select coalesce(sum(amount), 0)
from public.spending_resolutions
where user_id = auth.uid()
  and confirmed_at is not null;
```

本期不增加用户总额缓存字段。

## Agent 输出协议

复用现有回复末尾隐藏标记的方式：

- 建议购买：回复最后输出 `[decision:buy]`。
- 建议不买且金额明确：依次输出 `[decision:skip]` 和 `[spending_resolution:699.00]`。
- 普通追问、价格未知或尚未形成结论：不输出任何标记。

客户端只解析完整的 Assistant 回复。`spending_resolution` 金额必须是正数、最多两位小数，并落在数据库数值范围内；无效或缺失的金额不创建卡片。商品标题、链接、分类和图片由数据库函数从当前评估记录生成快照，不采信模型生成或客户端提交的商品信息。

隐藏标记不展示给用户。可见回复仍保存为正常的自然语言内容。

## 原子保存

新增 `save_evaluation_reply` 数据库函数，接收评估 ID、清理后的回复正文、决策和可选金额，原子完成以下操作：

1. 校验当前用户拥有目标购物评估。
2. 保存清理过隐藏标记的 Assistant 回复。
3. 根据本轮结论处理忍住记录：
   - 建议不买且金额有效：创建或更新未确认记录。
   - 建议买：删除该对话的未确认记录。
   - 没有最终结论：不改变忍住记录。
4. 若记录已经确认，拒绝任何更新或删除。

这样不会出现回复已经保存但卡片记录丢失的半完成状态。

确认使用 `confirm_spending_resolution` 数据库函数：它只允许记录所有者把目标记录的 `confirmed_at` 从空值更新为数据库当前时间。已确认请求再次到达时返回现有记录，不重复写入或累计。

## 聊天交互

卡片显示在触发动作的 Agent 回复下方，使用 Worth 现有颜色、圆角和间距，不复刻参考图。

待确认状态：

- 标题：`这次先不买`
- 金额：`留下 ¥699`
- 按钮：`确认不买`

点击后按钮显示加载状态。数据库确认成功后切换为：

- `已忍住 ¥699`
- 确认时间
- 不再显示操作按钮

确认失败时仍保留待确认状态，并显示“确认失败，请重试”。不做乐观累计。

卡片只在流式回复结束、回复和忍住记录原子保存成功后出现。重新进入历史对话或换设备后，客户端从 `spending_resolutions` 恢复待确认或已确认状态。

## 权限与约束

- 表启用 RLS，只允许已登录用户读取自己的记录。
- 记录创建、更新、撤销和确认通过数据库函数完成，并在函数内再次校验购物评估所有权。
- `evaluation_id` 唯一，阻止同一对话创建多条记录。
- 数据库约束校验正数金额和必要的商品快照。
- 已确认记录在数据库函数层不可变，客户端状态不作为可信依据。

## 验证

最小验证范围：

- 动作标记解析：有效整数、小数、缺失金额、负数、超精度和乱码。
- 数据库：RLS 所有权、唯一对话、未确认更新、结论变更撤销、确认后不可修改。
- 幂等：重复点击、请求重试和并发确认只产生一条已确认记录。
- 聊天组件：待确认、确认中、失败重试、已确认和历史恢复。
- 回归：运行现有服务端测试、移动端 Node 测试和 TypeScript 检查。

## 不在本期

- “忍住消费”总额的展示页面。
- 24 小时冷静期、通知和定时任务。
- 确认后的撤销、编辑或删除。
- 用户总额缓存、排行榜、成就或公益基金换算。
