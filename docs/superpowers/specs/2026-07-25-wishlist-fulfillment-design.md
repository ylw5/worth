# 心愿实现与资金分配设计

## 目标

为心愿清单增加“实现心愿”能力。用户实现心愿时填写实际成交价，并从“忍住消费”和“已卖闲置”的可用记录中选择资金来源。系统自动计算每条记录的使用金额和自付差额，保证同一笔资金不会被重复使用。

## 产品规则

- 心愿创建时的 `target_price` 是预期金额，继续用于未实现阶段的进度展示。
- 实现心愿时填写正数的实际成交价，默认带入预期金额。
- 用户可以按顺序勾选具体的“忍住消费”和“已卖闲置”记录。
- 客户端按勾选顺序分配资金：优先用完前面的可用余额，最后一条按需部分使用，达到实际成交价后停止。
- 一条资金记录可以拆分给多个心愿；可用余额为原金额减去所有已分配金额。
- 允许资金不足，未覆盖部分为自付金额。
- 允许不选择资金记录，此时实际成交价全部计为自付。
- 自付金额不单独存储，由实际成交价减去资金分配总额计算。
- 已实现心愿不再参与未实现心愿的进度展示。
- 用户可以撤销实现。撤销后删除该心愿的资金分配、恢复来源余额，并将心愿恢复为未实现。
- 已实现心愿不能直接删除；需要先撤销实现。
- 修改实际成交价或资金来源时，统一撤销后重新实现。

## 数据模型

在 `public.wishlist_items` 增加：

| 字段 | 类型 | 约束与用途 |
| --- | --- | --- |
| `actual_price` | `numeric(12, 2)` | 可空；实现后的实际成交价，非空时大于 0 |
| `fulfilled_at` | `timestamptz` | 可空；实现时间 |

两列必须同时为空或同时非空。

新增 `public.wishlist_funding_allocations`：

| 字段 | 类型 | 约束与用途 |
| --- | --- | --- |
| `id` | `uuid` | 主键，默认生成 |
| `user_id` | `uuid` | 必填，关联 `auth.users` |
| `wishlist_item_id` | `uuid` | 必填，关联心愿；使用删除限制 |
| `spending_resolution_id` | `uuid` | 可空，关联一条已确认的忍住消费记录 |
| `asset_sale_id` | `uuid` | 可空，关联一条闲置成交记录 |
| `amount` | `numeric(12, 2)` | 必填且大于 0 |
| `created_at` | `timestamptz` | 分配时间 |

每行必须且只能填写 `spending_resolution_id` 或 `asset_sale_id` 其中一个。同一个心愿不能重复引用同一来源。

资金来源使用删除限制。已被分配的闲置成交记录不能修改成交信息、删除或改回未卖出，用户需要先撤销对应心愿。

表启用 RLS，只允许已登录用户读取自己的分配记录。写入和删除只通过数据库函数完成。

迁移同时收紧现有心愿权限：

- 保留当前用户对自己心愿的读取、创建和未实现心愿删除能力。
- 已实现心愿不能直接删除，包括没有资金分配的全额自付心愿。
- 客户端不能直接更新 `actual_price` 和 `fulfilled_at`，两列只由确认和撤销函数写入。

## 原子操作

### 确认实现

新增 `fulfill_wishlist_item` 数据库函数，接收：

- 心愿 ID；
- 实际成交价；
- 客户端根据勾选顺序生成的资金分配数组，数组元素包含来源类型、来源 ID 和使用金额。

函数在同一事务内：

1. 锁定并校验当前用户拥有目标心愿，且心愿尚未实现。
2. 校验实际成交价为正数。
3. 拒绝重复来源、非正数分配金额和超过实际成交价的分配总额。
4. 锁定所有资金来源，校验来源属于当前用户。
5. 忍住消费来源必须已经确认。
6. 分别计算每条来源已分配金额，校验本次使用金额不超过真实可用余额。
7. 写入资金分配记录。
8. 保存心愿的实际成交价和实现时间。

若来源余额已被另一台设备使用，函数拒绝整个请求，不静默提高自付金额。客户端刷新余额后让用户重新确认。

重复提交已经实现的心愿时函数拒绝请求，不创建额外分配。

### 撤销实现

新增 `unfulfill_wishlist_item` 数据库函数。函数锁定并校验心愿所有权，删除该心愿的全部资金分配，并同时清空 `actual_price` 和 `fulfilled_at`。删除分配后，对应来源余额自然恢复。

未实现心愿收到撤销请求时函数拒绝请求，不改变数据。

## 查询与计算

心愿页读取：

- 心愿列表；
- 已确认的忍住消费记录及其已分配总额；
- 闲置成交记录及其已分配总额；
- 已实现心愿的资金分配明细。

每条来源的可用余额：

```text
原始金额 - 已分配金额
```

未实现心愿的公共进度改为使用两类来源的可用余额合计，避免已经实现其他心愿的资金继续出现在进度中：

```text
可用资金 = 忍住消费可用余额 + 已卖闲置可用余额
进度 = min(可用资金 / 预期金额, 100%)
```

已实现心愿的自付金额：

```text
自付金额 = 实际成交价 - 该心愿资金分配总额
```

金额由数据库精确数值返回，客户端沿用现有金额格式化方式。

## 页面交互

### 未实现心愿

心愿卡保留预期金额和可用资金进度，并增加“实现心愿”按钮。

点击后进入确认页：

1. 实际成交价输入框默认带入预期金额，允许修改。
2. 资金记录分为“忍住消费”和“已卖闲置”两组。
3. 只显示可用余额大于零的记录。
4. 每行展示记录名称、原金额、已使用金额和可用余额。
5. 用户按顺序勾选记录，页面实时生成抵扣明细。
6. 汇总区展示实际成交价、资金抵扣和自付金额。
7. 最后一条只需部分使用时，明确展示本次实际使用金额和剩余余额。

点击“确认实现”后调用原子数据库函数。成功后刷新心愿、资金来源和分配明细查询；失败时保留当前输入与选择，并显示错误。

### 已实现心愿

已实现心愿从未实现 Carousel 移到下方“已实现”区域。每项展示：

- 心愿名称；
- 实际成交价；
- 实现时间；
- 忍住消费抵扣合计；
- 已卖闲置抵扣合计；
- 自付金额；
- 可展开的具体资金来源和每条使用金额；
- “撤销实现”操作。

撤销前显示二次确认。成功后刷新相关查询，心愿回到未实现列表，资金余额同步恢复。

## 状态与错误处理

- 实际成交价为空、非数字或不大于零时不能提交。
- 没有选择资金来源时允许提交，并明确展示全额自付。
- 分配总额不会超过实际成交价。
- 确认或撤销期间禁用重复操作。
- 确认和撤销均不做乐观账目更新。
- 来源余额冲突时提示“资金余额已变化，请重新确认”，刷新来源后保留实际成交价。
- 网络或数据库失败时保持原账目状态，允许重试。
- 已实现心愿的直接删除由数据库外键限制阻止。

## 验证

最小验证范围：

- 客户端纯函数：按勾选顺序全额使用、最后一条部分使用、资金不足产生自付、零来源全额自付、已分配余额不再可用。
- 数据库：只接受正数实际成交价和分配金额。
- 数据库：只允许使用当前用户自己的已确认忍住记录和闲置成交记录。
- 数据库：同一来源可拆分给多个心愿，但累计分配不能超过原金额。
- 数据库：重复来源、分配总额超过实际成交价和重复实现均失败。
- 数据库：并发请求不能导致来源超额分配。
- 数据库：撤销后分配删除、余额恢复、心愿回到未实现。
- 数据库：跨用户确认、撤销和读取均被拒绝。
- 界面：确认成功、余额冲突、普通失败、已实现展示和撤销确认。
- 回归：运行相关 Node 测试、TypeScript 检查和代码格式检查。

## 不在本期

- 多次购买或部分实现状态。
- 直接编辑已实现账目。
- 自动推荐或自动选择资金来源。
- 资金转账、提现或真实支付能力。
- 成就、动画、通知或社交分享。
- 新增服务端应用接口；本功能沿用 Supabase 查询和数据库函数。
